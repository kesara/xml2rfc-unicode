1. Introduction

Non-ASCII characters are allowed in RFCs under several restrictions. See [RFC7997].¶

Some XML2RFC elements allow the use of bare Unicode and other elements can make use of >u< element. See [RFC7991].¶

Various non-Latin Unicode characters can be an issue in RFC publications. Especially generating PDF format of the document. This is because the correct font has to be identified and included in the PDF file. This is done by the xml2rfc too. See [xml2rfc].¶

xml2rfc maintains its own list of Unicode script blocks. Depending on which script is used, xml2rfc matches the correct Noto font.¶

This document is an experiment on when Unicode characters from different blocks are included in an Internet-Draft.¶

2. Methodology

The Unicode characters blocks are identified using [unicode-blocks].¶

This random printable character from each block is chosen. The xml2rfc is used as an API to identify which script blocks are assigned to that character and predicted font names. xml2rfc might predict a non-existing font. These are the instances where xml2rfc might leak non-standard fonts to generated PDF files.¶
